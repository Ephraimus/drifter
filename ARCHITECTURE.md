# 🏛️ Architecture & Pipeline Manual

Данный документ описывает внутреннее устройство Единой Консоли для Windows Terminal, WSL и Linux-серверов. Текст содержит структурное обоснование, стратегию миграции и регламенты безопасности развёртывания (Architectural Decision Record). 

👉 **Инструкции по быстрой установке доступны в файле [README.md](README.md).**

---

## 1. Архитектурная философия и Технологический стек

В основе проекта лежит концепция недеструктивного внедрения. Смена системной оболочки по умолчанию (через утилиты вроде `chsh` на `zsh` или `fish`) сопряжена с рисками нарушения целостности системных сценариев и скриптов инициализации ОС. Рекомендуемый стек строится на принципах отказоустойчивости и предсказуемости.

### 1.1 Ядро и Оболочка
- **Bash:** Основная оболочка для целевых машин. Гарантирует 100% совместимость со всеми POSIX-скриптами ОС (в отличие от Fish/Zsh). Используется как универсальный загрузчик среды.
- **Ble.sh (Bash Line Editor):** Фреймворк, внедряющий современные механизмы (автодополнения истории по мере ввода, подсветку синтаксиса до выполнения команд) непосредственно в `bash`, превращая его в современную интерактивную среду.
- **Fish (Опционально):** Рекомендуется к активации только на локальных рабочих станциях (например, WSL среды разработчика), где приоритетом является агрессивный пользовательский интерактив и богатый UX "из коробки", а не POSIX-совместимость.

### 1.2 Управление конфигурациями (Chezmoi)
**Chezmoi** выступает единым шаблонизатором и менеджером файлов конфигурации (dotfiles).
Главная причина выбора данного инструмента — способность гибко адаптироваться под целевую ОС (функции `lookPath`, шаблоны `if/else`). Chezmoi обеспечивает жесткое разделение глобальной экосистемы репозитория от локальных аппаратных исключений посредством локальных файлов данных (`chezmoi.toml`). Это предотвращает сбои на серверах.

### 1.3 Информационный интерфейс (Starship)
Вопреки трендам на перегруженные визуально темы, в качестве универсального генератора выбран **Starship**. Он написан на Rust, исполняется за миллисекунды, и, самое важное, *не зависит* от используемой командной оболочки, обеспечивая одинаковый UX как в Bash, так и в Fish.

### 1.4 Модернизация системных команд (Rust CLI)
Система заменяет классические, морально устаревшие системные утилиты UNIX на современные высокопроизводительные кроссплатформенные альтернативы:
- **`eza` (ex-exa)** заменяет `ls`. Предоставляет цвета, интеграцию с Git и иконки.
- **`bat` / `batcat`** заменяет `cat`. Предоставляет встроенную подсветку синтаксиса большинства ЯП.
- **`zoxide`** заменяет `cd`. Реализует интеллектуальную навигацию по директориям без написания полных путей.
- **`fzf`** — неопциональный стандарт для Fuzzy-поиска истории и файлов в пайплайне.

## 2. Структура конфигурационного хранилища `chezmoi`

```text
~/.local/share/chezmoi/
  dot_bashrc.tmpl               # Входной интерактивный скрипт инициализации bash
  dot_config/
    shell/
      env.list                  # Единый изолированный список переменных окружения
      aliases.sh.tmpl           # Центр инициализации алиасов оболочки
    fish/
      config.fish.tmpl          # Опциональный локальный Fish-конфиг
    starship.toml               # Конфигурация генератора prompt
  scripts/
    install.sh                  # Обертка для развертывания Bare Metal (curl-to-bash)
    bootstrap.sh                # Локальный инициализатор зависимостей
    bootstrap-remote.sh         # Сценарий доставки конфигураций по SSH
```

### 2.1 Единый источник истины (Single Source of Truth)
Для минимизации дублирования конфигураций между Bash, Zsh и Fish, базовые переменные и алиасы платформы вынесены в файлы `env.list` и `aliases.sh.tmpl`. Логика загрузки целевых оболочек запрограммирована на парсинг данных файлов при старте сессии.

### 2.2 Механизм адаптивной деградации (Fallback)
Шаблон `aliases.sh.tmpl` содержит логику верификации бинарных файлов через функцию шаблонизатора `lookPath`. При развертывании инфраструктуры на устаревших серверах, где невозможна установка новых утилит, конфигурация алиасов не приведет к ошибке `command not found`, а будет автоматически сгенерирована для использования системных `ls` и `cat`.

## 3. Регламенты внедрения и безопасности

Сценарии инсталляции (`bootstrap.sh` и `bootstrap-remote.sh`) спроектированы таким образом, чтобы полностью исключить деструктивные операции с пользовательскими данными.

### 3.1. Стратегия резервного копирования
Транзакция инициализации полностью копирует оригинальные файлы пользователя (`~/.bashrc`, `~/.zshrc`, `~/.config/fish/config.fish`, `starship.toml`) в изолированную директорию с защитным timestamp:
`~/.shell-migration-backup/YYYYMMDD-HHMMSS/`

### 3.2. Мягкий сброс (Soft Reset)
Данный режим применяется по умолчанию. Пайплайн переименовывает существующие конфигурационные файлы пользователя с расширением `*.legacy` (`~/.bashrc` -> `~/.bashrc.legacy`). 
Вновь сгенерированный шаблон `~/.bashrc` содержит специальную инструкцию `source ~/.bashrc.legacy`.
**Результат:** Все пользовательские скрипты, специфичные алиасы серверов и кастомные переменные продолжают функционировать, при этом визуальная и структурная инициализация производится в рамках внедряемого пайплайна (надстройка).

### 3.3. Жёсткий сброс (Hard Reset)
Активируется исключительно явным вызовом скрипта `./scripts/hard-reset-shell-configs.sh` или аргументом `--reset-mode hard`. Применяется в случаях, когда текущие системные конфигурации (например, перегруженный плагинами `oh-my-zsh` или конфликтующий кастомный prompt) нарушают стабильность инициализированной среды. Жёсткий сброс полагается исключительно на механизм резервного копирования, превентивно удаляя все симлинки шаблонизатора.

## 4. Специфичные локальные конфигурации (Machine-Specific)

Для предотвращения перезаписи локальных серверных исключений едиными глобальными правилами репозитория, на целевой машине генерируется индивидуальный конфигурационный файл `~/.config/chezmoi/chezmoi.toml`.

```toml
[data]
use_fish = false
use_blesh = false      # Допускает отключение на слабых серверах
legacy_zsh = true
```
Редактирование данного файла и последующий вызов `chezmoi apply` позволяют точечно управлять поведением пайплайна на конкретном сервере, не внося изменений в глобальный репозиторий.

---

## 5. Процедура отката конфигураций (Rollback)

Для полной деинсталляции пайплайна и возврата терминала к первичному состоянию предусмотрен следующий алгоритм:
1. Перейдите в директорию склонированного репозитория (или `~/.local/share/chezmoi/`).
2. Выполните скрипт `./scripts/hard-reset-shell-configs.sh` для автоматического удаления симлинков.
3. Восстановите оригинальные файлы из папки резервного копирования `~/.shell-migration-backup/последняя_дата/` в домашнюю директорию `~/`.

## 6. Базовые настройки конфигураций (Anatomy of Defaults)

Каждая строка в конфигурациях инфраструктуры внедрена с учетом решения профильных эксплуатационных задач инженеров. Данный раздел описывает логику принятия решений по дефолтным параметрам.

### 6.1. Пользовательский интерфейс: Starship
Файл: `~/.config/starship.toml`

Генератор `starship` настроен в рамках строгой минималистичной парадигмы. Вместо постоянной перегруженной индикации нерелевантной информации, используются модули реактивного состояния (Debugger Mode):
- **`[status]`**: Модуль вывода активируется исключительно в случае ненулевого (ошибочного) кода возврата предыдущей команды (вызывая появление индикатора `[✘ 127]`). Упрощает отладку скриптов.
- **`[jobs]`**: Индикатор `✦ 1` визуализируется при наличии приостановленных фоновых процессов (например, свернутых через `Ctrl+Z`). Предотвращает случайную потерю несохраненных данных при закрытии SSH-сессии.
- **`[sudo]`**: Индикатор `⚡` уведомляет администратора об активных правах root или авторизованной кэшированной сессии суперпользователя в рамках защиты от непреднамеренных инцидентов.

### 6.2. Адаптивная деградация (Фоллбэки в Chezmoi)
Файл: `~/.config/shell/aliases.sh.tmpl`

Шаблонизатор `chezmoi` использует функции `lookPath` для верификации зависимостей перед генерацией итоговых конфигурационных файлов:

```gotemplate
{{- if lookPath "eza" }}
alias ls='eza --icons=auto --color=auto'
alias ll='eza -lh --icons=auto'
{{- else }}
alias ll='ls -lah'
{{- end }}
```
**Принцип действия:** При деплое конфигурации на сервер, лишенный утилиты `eza`, шаблонизатор генерирует файл с POSIX-эквивалентом `ls -lah`, предотвращая регулярный выброс системных исключений `command not found` во время загрузки оболочки.

### 6.3. Интеграция превьюверов в Fuzzy-поиск (FZF + Bat)
Файл: `~/.config/shell/env.list`

Логика встроенного инструмента навигации `FZF` (Fuzzy Finder) расширена глобальной переменной, предоставляющей возможность предварительного динамического просмотра файлов с подсветкой синтаксиса:
```env
FZF_DEFAULT_OPTS=--preview 'bat --color=always {} 2>/dev/null || batcat --color=always {} 2>/dev/null || cat {}'
```
Последовательная инициализация пайплайна через оператор `||` эффективно предотвращает сбои зависимости:
1. Вызов утилиты `bat` (де-факто наименование для macOS Brew и Arch Linux).
2. Вызов утилиты `batcat` (стандартизированное наименование в экосистеме Debian/Ubuntu).
3. При отсутствии компонентов инициируется fallback-сценарий через системный вызов `cat`.

### 6.4. Базовые принципы встроенных утилит навигации
- **`zoxide`**: Механизм навигации, учитывающий частотностью посещений директорий. Допускает использование предиктивного псевдонима `z proj` вместо полных абсолютных путей. Ускоряет переход по проектам.
- **`ble.sh`**: Фреймворк Bash Line Editor, обеспечивающий динамическую валидацию пользовательского ввода (подсветка синтаксиса перед выполнением энтерпретатором) и автодополнения истории исполнения, по аналогии с логикой оболочек Zsh-autosuggestions и Fish.

## 7. Жизненный цикл инфраструктуры (Workflow & Secrets Management)

Управление репозиторием базируется на механиках Git и Chezmoi для безопасного и консистентного распространения обновлений конфигураций по рабочим станциям.

### 7.1. Редактирование локальных конфигураций (`chezmoi edit`)
**Ограничение:** Прямое редактирование генерируемых файлов в домашней директории (например, `~/.config/shell/aliases.sh`) приведет к их перезаписи во время последующей синхронизации шаблонизатора (и потере внесенных данных).
**Процедура редактирования:** Внесение легитимных изменений производится строго через команду редактирования шаблонов:
```bash
chezmoi edit ~/.config/shell/aliases.sh
```
Данная команда открывает исходный **шаблон** в защищенной локальной директории репозитория (`~/.local/share/chezmoi/`). После сохранения изменений необходимо выполнить `chezmoi apply` для трансляции конфигураций в пользовательскую операционную систему.

### 7.2. Публикация и Синхронизация версий (`chezmoi update`)
Локальный инстанс репозитория `chezmoi` функционирует как стандартное хранилище Git. Описание пайплайна синхронизации:
1. Перейдите в системную директорию хранилища: `cd ~/.local/share/chezmoi`
2. Зафиксируйте внесенные изменения: `git add . && git commit -m "Update alias definitions" && git push`
3. Для инициализации обновлений на удаленной машине или сервере выполните:
```bash
chezmoi update
```
**Принцип транзакции:** Шаблонизатор клонирует апдейт из удаленного Git-репозитория и перегенерирует скрипты, повторно применяя адаптивную логику функции `lookPath` к конкретной целевой машине (масштабирование без ошибок).

### 7.3. Управление Секретами (API-ключи и авторизационные Токены)
**Требование информационной безопасности:** Запрещается кодировать (хардкодить) токены (Github PAT, AWS IAM Keys, OpenAI API) в глобальных файлах репозитория `env.list`, так как это ведет к критической компрометации данных в системах контроля версий.

Допустимые подходы к хранению учетных данных в рамках инфраструктуры:
1. **Интеграция с Password Managers:** В шаблонах переменных поддерживается функционал извлечения паролей через интегрированные CLI-утилиты сторонних менеджеров (Bitwarden, KeePassXC, `pass`, Vault). 
Пример синтаксиса для `env.list.tmpl`:
`export AWS_KEY={{ (bitwarden "itemid").notes }}`
2. **Изолированный неиндексируемый `.env` (Точечный подход):**
Предполагает создание файла `~/.env.private` на целевом сервере. Имя данного файла строго зафиксировано в глобальном `.gitignore`. 
Шаблон инициализатора `dot_bashrc.tmpl` содержит условный импорт переменных:
```gotemplate
[ -f "$HOME/.env.private" ] && . "$HOME/.env.private"
```
Данный подход обеспечивает надежную изоляцию уникальных секретных токенов для конкретной машины или контейнера без риска их загрузки в единый Git-репозиторий организации.

## 8. Синхронизация удалённых хостов

### 8.1. Стандартный сценарий (один сервер)
При наличии одного или двух серверов рекомендуется ручная синхронизация непосредственно на целевой машине:
```bash
chezmoi update
```
Данная команда клонирует последний коммит из настроенного Git-репозитория и перегенерирует конфигурации с учётом особенностей данного хоста (machine-specific `chezmoi.toml`).

### 8.2. Массовая синхронизация (`sync-all-remotes.sh`)
Для работы с несколькими серверами в репозиторий включён скрипт-оркестратор `scripts/sync-all-remotes.sh`. Он итерируется по массиву `HOSTS` и запускает `chezmoi update` через SSH параллельно.

**Конфигурация (редактировать сам скрипт):**
```bash
HOSTS=(
  "user@server1.example.com"
  "user@server2.example.com"
)
```

**Запуск:**
```bash
./scripts/sync-all-remotes.sh
```
По завершению скрипт выводит итоговую сводку с перечнем успешно синхронизированных хостов и хостов, завершившихся с ошибкой.

**Предварительные требования:**
- На всех целевых хостах должен быть установлен и инициализирован `chezmoi` (через `bootstrap-remote.sh`).
- SSH-ключ аутентификации без пароля должен быть добавлен на все целевые машины (`ssh-copy-id user@host`).

## 9. Справочник Команд: Полный перечень вариантов запуска скриптов

Данный раздел содержит все поддерживаемые варианты вызова скриптов установки и синхронизации с описанием каждого. Команды готовы к копированию.

---

### 9.1. `bootstrap.sh` — Локальная установка

Используется, если репозиторий уже клонирован на целевой машине.

**Минимальный запуск (только Bash + Starship):**
```bash
./scripts/bootstrap.sh
```

**Установка с активацией Fish-оболочки и сменой login shell:**
```bash
./scripts/bootstrap.sh --use-fish --set-login-shell fish
```

**Форсированная переустановка без сохранения legacy-конфигов:**
```bash
./scripts/bootstrap.sh --reset-mode hard -y
```

**Установка без пакетов (только конфиги, на уже настроенной системе):**
```bash
./scripts/bootstrap.sh --skip-packages
```

**Установка без бекапа и без сброса (idempotent, только поверх):**
```bash
./scripts/bootstrap.sh --skip-backup --skip-reset
```

**Установка из удалённого Git-репозитория (вместо локального каталога):**
```bash
./scripts/bootstrap.sh --repo-url git@github.com:Ephraimus/drifter.git --repo-branch main
```

**Все доступные флаги:**
| Флаг | Описание |
|---|---|
| `--source-dir DIR` | Путь к каталогу starter-kit (по умолчанию: родительская папка скрипта) |
| `--repo-url URL` | Альтернатива: использовать удалённый Git-репозиторий |
| `--repo-branch BRANCH` | Ветка для `--repo-url` |
| `--use-fish` | Установить Fish и активировать конфиг |
| `--reset-mode soft\|hard` | Режим сброса конфигов (по умолчанию: `soft`) |
| `--set-login-shell bash\|fish\|zsh\|skip` | Изменить login shell после установки |
| `--skip-packages` | Не устанавливать системные пакеты |
| `--skip-backup` | Не создавать резервную копию |
| `--skip-reset` | Не переименовывать существующие конфиги |
| `-y, --yes` | Не-интерактивный режим (обязателен для `hard` reset) |

---

### 9.2. `bootstrap-remote.sh` — Удалённая установка по SSH

Используется для доставки конфигурации на удалённый сервер без предварительного клонирования репозитория.

**Способ 1: Прямое подключение по SSH с передачей скрипта через `stdin`:**
```bash
cat ./scripts/bootstrap-remote.sh | ssh user@host 'bash -s -- --repo-url https://github.com/Ephraimus/drifter.git --repo-branch main'
```

**Способ 2: Использование локально скопированного каталога starter-kit:**
```bash
# Скопируем starter-kit на сервер
scp -r ./ user@host:/tmp/drifter-kit

# Запустим установку с удалённого каталога
ssh user@host 'bash /tmp/drifter-kit/scripts/bootstrap-remote.sh --starter-kit-dir /tmp/drifter-kit'
```

**Установка с активацией Fish на удалённом сервере:**
```bash
cat ./scripts/bootstrap-remote.sh | ssh user@host 'bash -s -- --repo-url https://github.com/Ephraimus/drifter.git --use-fish --set-login-shell fish'
```

**Полностью неинтерактивная установка в CI/CD pipeline:**
```bash
cat ./scripts/bootstrap-remote.sh | ssh -o StrictHostKeyChecking=no deploy@host 'bash -s -- --repo-url https://github.com/Ephraimus/drifter.git --yes --skip-backup'
```

### 9.3. `sync-all-remotes.sh` — Массовая синхронизация

Используется для применения изменений из Git на все настроенные серверы сразу.

**Шаг 1: Настройте список серверов в скрипте:**
```bash
# Откройте scripts/sync-all-remotes.sh и отредактируйте массив HOSTS:
HOSTS=(
  "user@server1.example.com"
  "user@server2.example.com"
)
```

**Шаг 2: Запустите синхронизацию:**
```bash
./scripts/sync-all-remotes.sh
```

**Или разовая синхронизация одного сервера без скрипта:**
```bash
ssh user@host 'chezmoi update'
```
